Problema de Negócio

Churn em Telecom

Operadora perde clientes por cancelamento, impactando receita recorrente e aumentando o custo de aquisição de novos clientes.

O Desafio

Com 7.043 clientes e 21 variáveis de comportamento disponíveis, a empresa não conseguia identificar proativamente quem tinha risco de sair.

Stakeholders

  • Marketing/CRM — campanhas de retenção segmentadas
  • Customer Success — priorização de contato
  • Financeiro — medir ROI e impacto em receita

Distribuição de Churn na Base

A Responsabilidade

Construir uma solução fim a fim de ML Engineering — não apenas treinar um modelo, mas entregar algo operacional.

🔍 Qualidade de Dados

Entender inconsistências, valores faltantes e a distribuição desbalanceada de churn na base.

💰 Modelagem Econômica

Não era sobre maximizar AUC, mas sobre maximizar o Net Benefit financeiro sob premissas de custo real.

⚙ Engenharia de Produção

Transformar experimentos em API robusta, com MLflow, testes automatizados e Docker.

📄 Comunicação Técnica

Documentar decisões, métricas e limitações para múltiplos públicos (ML Canvas, Model Card, EDA).

Execução: Dados & Modelagem
1

EDA e Limpeza de Dados

Análise exploratória no Jupyter: distribuições, correlações e comportamento de churn. Base com 26,5% de churn — dataset desbalanceado (razão 1:2,77). Pipeline de limpeza: valores faltantes, encoding de categóricas, feature engineering de tenure e tipo de contrato.

2

Modelagem Orientada por Negócio

Premissas definidas com o time: cost_fp = $50 (campanha desnecessária) · cost_fn = $2.000 (cliente perdido).

8 modelos treinados em paralelo: Baseline, Logistic Regression (3 variantes), Random Forest, XGBoost (tuned), MLP PyTorch. Para cada um, curva de Net Benefit × Threshold.

🏆 Vencedor: LogisticRegression-balanced — $706.000 de Net Benefit no threshold 0,15

* Estimativa baseada na média de permanência de 32,4 meses (~3 anos) dos clientes atuais — custo de churn de $2.000 por cliente perdido.

Execução: Infraestrutura & API
3

MLflow + Docker Compose

Rastreamento completo dos 8 modelos (hiperparâmetros, métricas, artefatos) com backend em PostgreSQL. Arquitetura com 4 serviços containerizados: FastAPI · MLflow UI · PostgreSQL · Training job

4

API de Produção FastAPI

3 endpoints: POST /api/predict · POST /api/predict-batch · GET /api/model-info. Validação Pydantic, tratamento de erros estruturado e health checks automáticos.

5

Testes & Documentação

Testes pytest (dados, modelo, API, edge cases) + smoke tests Postman. Documentação: Model Card · ML Canvas · Relatório EDA · Guias operacionais de deploy.

Aprendizado-chave

Otimizar AUC ≠ otimizar negócio.

O modelo com melhor Net Benefit não era o líder em F1-score — a decisão correta exigiu pensar pelo custo real do erro.
